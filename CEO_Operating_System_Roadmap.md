## 3. CEO Dashboard 应该如何设计

### 3.1 当前状态评估

当前 `01_CEO_Dashboard.md` 是一个**手动填写的Markdown模板**。所有数据字段都是TODO。优点是结构完整、覆盖维度全面；缺点是没有任何自动化，CEO需要手动汇总所有数据填入。

### 3.2 设计原则

1. **实时 > 定期:** 数据应自动汇聚，而非等CEO手动填入
2. **例外管理:** 正常项不显示，只显示需要关注的红/黄灯项
3. **一个屏幕:** CEO应该能在30秒内读完整个Dashboard
4. **可下钻:** 每个数字都应该能点击查看明细
5. **AI驱动:** AI员工应负责生成Dashboard，CEO只做审核和决策

### 3.3 三阶段演进路线

#### Phase 1: 手动但完整 (当前 — 立即使用)

当前的Markdown模板设计合理。**立即行动:** CEO从现在开始每周手动填写真实数据。即使是0也要写入，建立数据纪律。

```
优势: 无需任何技术投入
劣势: 手动耗时、无实时性
适用: 0-5个客户阶段
```

#### Phase 2: AI辅助半自动 (第2-6周)

部署CEO Assistant AI员工。工作流:

```
原始数据源 (邮件/日历/微信/CRM) 
    → AI CEO Assistant 汇总 
    → 自动生成Dashboard草稿 
    → CEO审核修改 
    → 发布到共享位置
```

**数据源集成:**
- Google Sheets作为轻量数据库
- AI通过API读取邮件、日历
- 微信聊天记录手动输入关键信息

**技术方案:**
```
Google Sheets (结构化数据存储)
    ↕ API读写
Claude Project (AI CEO Assistant)
    → 读取: CEO_Operating_System/*.md
    → 读取: Google Sheets (pipeline, financials, projects)
    → 输出: 每周一9:00生成Dashboard摘要
    → 输出: 每日8:00推送当天优先级
```

#### Phase 3: 实时Dashboard (第7-12周)

当客户数达到5+且团队扩大时，迁移到轻量级BI工具:

**推荐方案:**
- **Google Looker Studio** (免费, 连接Google Sheets)
- 或 **Notion Database + Charts** (如果已经在用Notion)
- 或 **Metabase** (开源, 自部署)

**Dashboard应该包含的7个面板:**

| 面板 | 更新频率 | 数据源 |
|------|:--------:|--------|
| ① 收入概览 | 每日 | 银行/Stripe → Sheets |
| ② Pipeline健康度 | 每日 | CRM工具 |
| ③ 项目状态 | 每周 | 项目管理工具 |
| ④ 客户满意度 (NPS) | 每月 | 调查结果 |
| ⑤ AI员工效率 | 每周 | AI使用量统计 |
| ⑥ 现金流 | 每周 | 银行账户 |
| ⑦ 关键风险 | 每周 | 项目管理工具 |

| **周一 10:00** | 检查Pipeline → 标记本周需跟进的线索 | AI Sales + CEO | 15分钟 |

**CEO每周在Dashboard上的总时间: ~2小时** (Phase 1手动期) → **~30分钟** (Phase 2 AI辅助期)

## 4. CRM 应该如何设计

### 4.1 当前状态评估

`CRM-System.md` 已经包含了一个设计优秀的32字段Schema和10阶段状态机。但它是Markdown文件，无法做自动化、搜索、筛选、提醒。

### 4.2 CRM 三阶段实施路线

#### Phase 1: Google Sheets (第1-30天, 0-20个客户)

**为什么Google Sheets而非继续用Markdown:**
- 可筛选、排序、搜索
- AI可以通过Google Sheets API直接读写
- 可以设置条件格式 (红色=超期跟进)
- 免费，零学习成本
- 可以导出为任何格式

**Sheet结构设计 (5个Tab):**

```
CRM_Master_Sheet
├── Tab 1: Pipeline (活跃线索)
│   └── 32个字段 × N条记录
│   └── 条件格式: P0红色/P1橙色/P2蓝色/P3灰色
│   └── 筛选器: 按阶段、优先级、负责人、国家
│
├── Tab 2: Clients (已签约客户)  
│   └── 包含: NPS、LTV、续约日期、健康度
│
├── Tab 3: Contacts (联系人库)
│   └── 每个联系人的独立记录
│   └── 关联到公司
│
├── Tab 4: Activities (活动日志)
│   └── 每次会议、邮件、电话的记录
│   └── 自动时间戳
│
└── Tab 5: Reports (自动报表)
    └── Pipeline漏斗图
    └── 阶段转化率
    └── 本月新增/成交/流失
```

**AI集成:**
- AI Sales Manager 通过 Google Sheets API 读写Pipeline
- 每日自动检查: 是否有超期未跟进的线索? → 推送通知到CEO
- 每周一自动生成: Pipeline健康报告

#### Phase 2: Notion Database (第31-90天, 当客户达10-50个)

**升级触发条件:** 以下任一满足时从Sheets迁移到Notion
- 活跃线索+客户超过30条
- 需要多人协作 (不同人拥有不同记录)
- 需要更丰富的字段类型 (文件上传、关联表、多视图)

**Notion CRM结构:**

```
CRM (Notion Database)
├── 主表: Companies (公司)
│   ├── 关联: Contacts (联系人表)
│   ├── 关联: Deals (交易表)  
│   ├── 关联: Projects (项目表)
│   └── 关联: Activities (活动日志表)
│
├── 视图:
│   ├── Pipeline Board (看板视图, 按阶段分组)
│   ├── Priority Queue (列表视图, 按优先级排序)
│   ├── Calendar (日历视图, 按跟进日期)
│   └── CEO Summary (画廊视图, 仅显示P0/P1)
│
└── 自动化 (Notion Automations):
    ├── 跟进日期到期 → Slack通知
    ├── 7天无活动 → 自动标记⚠️
    └── 阶段变更 → 自动记录时间戳
```

#### Phase 3: 专业CRM (90天+, 客户>50或团队>5人)

**评估标准:**
- 必须有API (AI集成是首要选择标准)
- 支持自定义字段和Pipeline阶段
- 邮件/日历集成
- 免费/低价起步

**候选工具:**
| 工具 | 免费版 | API | 适合度 |
|------|:------:|:---:|:------:|
| **HubSpot** | 有(有限) | 好 | ⭐⭐⭐⭐⭐ |
| **Pipedrive** | 14天试用 | 好 | ⭐⭐⭐⭐ |
| **Attio** | 有(有限) | 好 | ⭐⭐⭐⭐ |
| **Twenty** (开源) | 免费 | 好 | ⭐⭐⭐ |

**推荐: HubSpot免费版** — 功能足够前50个客户使用，API完善，AI可深度集成。

### 4.3 CRM 自动化流程图

```
线索来源
├── 网站联系表单 → 自动创建CRM记录 → AI Sales自动回复
├── WeChat消息 → 手动/半自动录入 → AI辅助跟进
├── LinkedIn咨询 → 手动录入 → AI辅助跟进
├── 客户推荐 → 手动录入 (标记推荐来源)
└── 会议/展会 → 手动批量导入

CRM记录创建后:
  → AI Sales: 研究客户背景，写个性化外联草稿
  → CEO审核 → 发送
  → 自动跟踪: 3天无回复 → AI生成跟进提醒
  → 7天无回复 → 第二次跟进
  → 14天无回复 → 第三次跟进 (最后尝试)
  → 21天无回复 → 自动移至LOST (3个月后重新激活提醒)

签约后:
  → 自动创建项目记录
  → AI Ops: 生成onboarding checklist
  → 自动关联合同和发票
```

### 4.4 CRM 数据质量规则

| 规则 | 说明 |
|------|------|
| **每条记录必须有Owner** | 不允许孤儿记录存在 |
| **每个阶段必须有时限** | Lead 3天, Qualified 7天, Meeting 3天, Proposal 7天, Negotiation 14天 |
| **超时自动升级** | 超时未推进自动标记+通知CEO |
| **Lost必须记录原因** | 不可空白，用于分析和改进 |
| **每周一全Pipeline审查** | CEO + AI Sales一起过一遍 |

## 5. Business Operating System 应该如何设计

### 5.1 定义

Business Operating System (BOS) = 公司日常运营所需的所有系统、流程、工具和节奏的集合。它让公司"自动驾驶"，CEO只处理例外和战略。

### 5.2 BOS架构总览

│              ThaiBridge Business Operating System             │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌─────────────┐ │
│  │ 客户获取  │  │ 项目交付  │  │ 财务管理  │  │ AI Workforce │ │
│  │          │  │          │  │          │  │             │ │
│  │ • 网站   │  │ • 项目   │  │ • 开票   │  │ • 9个AI角色 │ │
│  │ • CRM   │  │   Tracker│  │ • 收款   │  │ • 知识库    │ │
